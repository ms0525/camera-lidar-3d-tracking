# Camera/LiDAR Detection and Tracking Prototype

This project runs an Ultralytics YOLO 2D detector on KITTI camera images,
optionally assigns temporal IDs with Deep SORT, and associates projected LiDAR
returns with each 2D box to estimate an approximate 3D location for Open3D
visualization. The runners accept an explicit YOLO checkpoint, including
YOLO11 and YOLO26, while retaining compatibility with the earlier YOLOv8
experiments.

It is deliberately described as a **prototype**, not a learned 3D detector.
Object dimensions are class priors, yaw is not estimated, and point association
inside a 2D box can be ambiguous when objects overlap.

The public repository contains source code, synthetic portfolio assets, and a
compact metrics summary. It intentionally excludes KITTI data, model weights,
raw experiment outputs, local screenshots, and machine-specific files. The
hosted Streamlit mode is synthetic; real inference requires a local clone plus
data and compatible checkpoints supplied by the user.

## Pipeline

```text
camera image -> YOLO 2D boxes -> Deep SORT IDs
                         |
calibration + LiDAR -----+-> point projection / depth clustering
                                  |
                                  +-> approximate 3D boxes in Open3D
```

KITTI projection uses the complete transform:

```text
pixel ~ P2 @ R0_rect @ Tr_velo_to_cam @ point_velodyne
```

The implementation also accounts for the camera-2 offset encoded in `P2` and
places rays at the actual camera center in the Velodyne frame.

Deep SORT association is class-aware in both its appearance and fallback-IoU
matching stages. A track can match only the same normalized detector class;
for example, a `car` track cannot silently become `person`, `van`, or `truck`.

## Installation

The original CPU workflow was tested with Python 3.10. The native-Windows AMD
training workflow below was tested with Python 3.12.10.

```powershell
python -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --upgrade pip
pip install -r requirements.txt
```

`requirements.txt` pins `ultralytics==8.4.95`. Use that version for the
documented YOLO26 head selection and for comparable experiment manifests.
Install the PyTorch build for the intended backend (ROCm, CUDA, or CPU) before
installing this common requirements file.

The root dependency set also pins `setuptools==80.10.2`. Deep SORT 1.3.2 still
imports the legacy `pkg_resources` module, which Setuptools removed in version
82; allowing an unconstrained upgrade makes the appearance embedder fail at
startup.

The repository's old `venv/` directory is a machine-specific generated
environment and should not be copied or committed. It is now excluded by
`.gitignore`.

### Native Windows AMD ROCm setup

The tested AMD environment uses an RX 7600S (`gfx1102`), ROCm 7.13,
`torch==2.9.1+rocm7.13.0`, and Ultralytics 8.4.95. ROCm 7.13 is currently a
preview SDK; use the matching packages from AMD's index and keep this
environment separate from the existing CPU environment. See AMD's
[ROCm 7.13 installation guide](https://rocm.docs.amd.com/en/7.13.0-preview/install/rocm.html)
and [Windows prerequisites](https://github.com/ROCm/TheRock/blob/main/docs/development/windows_support.md).

```powershell
py -3.12 -m venv "C:\path\to\3D_Detection_ROCm"
$rocmPython = "C:\path\to\3D_Detection_ROCm\Scripts\python.exe"

& $rocmPython -m pip install --upgrade pip
& $rocmPython -m pip install `
  --index-url "https://repo.amd.com/rocm/whl/gfx110X-all/" `
  torch==2.9.1+rocm7.13.0 `
  torchvision==0.24.0+rocm7.13.0 `
  torchaudio==2.9.0+rocm7.13.0 `
  "rocm[libraries,devel]==7.13.0"
& $rocmPython -m pip install -r requirements.txt
& $rocmPython -m pip check
```

Visual Studio 2022 v143 C++ build tools are also required for MIOpen's runtime
kernel compiler. On this laptop, an incomplete newer MSVC directory existed
beside the complete 14.40 toolset. The maintained launcher filters candidates
for the actual `type_traits` header and adds the matching ROCm Clang resource
headers, so no system-wide `INCLUDE` change is made.

The `rocm-sdk-devel` wheel contains a large lazily expanded archive. Eager
`rocm-sdk init` uses hard links and therefore fails on an exFAT environment
drive. Eager expansion is not required by the launcher. If SDK development
files are needed outside Python, place that environment on NTFS or allow about
25 GiB for a compatible expansion method.

PyTorch deliberately exposes ROCm devices through its `torch.cuda` API. It is
therefore normal for Ultralytics to print `CUDA:0 (AMD Radeon RX 7600S)` and for
the command to use `--device 0`; `torch.version.hip` and the run manifest's
`hardware.backend` distinguish ROCm from CUDA. See PyTorch's
[HIP semantics documentation](https://docs.pytorch.org/docs/stable/notes/hip.html).

## Four-panel Streamlit dashboard

The browser dashboard keeps one synchronized frame index across four views:

1. YOLO11 tracked detections in the camera image.
2. YOLO11 LiDAR-supported approximate 3D tracks.
3. YOLO26 tracked detections in the camera image.
4. YOLO26 LiDAR-supported approximate 3D tracks.

Use the Left and Right arrow keys or the on-screen controls to move through the
sequence. Autoplay advances the same shared index, can loop, and stops when a
non-looping sequence reaches its last frame. Each model owns one independent
Deep SORT timeline; a model runs only once for a new frame and its cached
`FrameAnalysis` feeds both corresponding panels. Going backwards does not run
inference again.

In live mode, the frame slider exposes cached frames plus exactly one new frame
ahead. This makes the temporal cost explicit and prevents an accidental jump
to a distant frame from launching hundreds of synchronous detector passes.

The default app is a lightweight, deterministic synthetic preview. It is
designed for a public portfolio host and is clearly identified as illustrative,
not as KITTI output or measured YOLO performance:

```powershell
& "C:\path\to\3D_Detection_ROCm\Scripts\python.exe" `
  -m pip install -r app\requirements.txt
& "C:\path\to\3D_Detection_ROCm\Scripts\python.exe" `
  -m streamlit run app\streamlit_app.py
```

For real local inference with the full KITTI Tracking sequence and both model
checkpoints, use the validated ROCm launcher:

```powershell
.\scripts\run_streamlit_rocm.ps1 `
  -DatasetRoot "C:\path\to\KITTI\tracking" `
  -Sequence 0000 `
  -Yolo11Model ".\yolo11s.pt" `
  -Yolo26Model ".\yolo26s.pt"
```

Live mode is intentionally opt-in and dynamically imports the heavyweight
Ultralytics, Deep SORT, OpenCV, and Open3D stack. Public preview mode does not
download weights, access a local drive, or redistribute KITTI data. Deploy
`app/streamlit_app.py` from GitHub on Streamlit Community Cloud; its dependency
resolver will use the adjacent `app/requirements.txt`. Full setup, environment
variables, hosting limits, and licensing checks are documented in
[`app/DEPLOYMENT.md`](app/DEPLOYMENT.md).

The ROCm launcher enables live mode with two explicit environment guards and
binds the server to `127.0.0.1`. Do not expose its editable filesystem paths as
an unauthenticated public service; a future public inference backend should use
authentication and fixed asset identifiers.

The web 3D panels preserve native Velodyne axes (`x` forward, `y` left, `z` up)
and use display-only point decimation. As in the Open3D viewer, predicted 3D
centers come from projected LiDAR depth clusters, dimensions are class priors,
and yaw is not estimated. Magenta ground-truth boxes are exact label geometry
when labels are available; they should not be confused with predicted boxes.

## Local sample data (not distributed)

No KITTI image, point cloud, calibration, or annotation is distributed in this
repository. Download KITTI from its official website under its own terms. For
the single-frame command examples, place one internally matched sample at
`data/images/000000.png`, `data/calib/000000.txt`, and
`data/pointcloud/000000.bin`, or use other explicit paths. These directories
are ignored by Git; the CLI tools require the paths rather than silently
assuming that private samples exist.

Independent object-detection samples are useful only for loader, projection,
and single-frame smoke tests. They are not a temporal sequence and must not be
used to demonstrate or evaluate tracking; persistent IDs require a real KITTI
Tracking sequence.

Export one complete sequence in official KITTI Tracking result format:

```powershell
python track_sequence.py `
  --dataset-root "C:\path\to\KITTI\tracking" `
  --sequence 0000 `
  --model yolo26s.pt `
  --imgsz 640 `
  --yolo-end2end `
  --headless `
  --export-kitti data/kitti_predictions/0000.txt
```

Export is deliberately restricted to complete headless runs. Each row has all
18 KITTI result fields, including the matched detector confidence. Only
`car -> Car` and either `person -> Pedestrian` (COCO checkpoints) or
`pedestrian -> Pedestrian` (the eight-class KITTI fine-tuned checkpoints) are
exported because those are the classes evaluated by the current KITTI Tracking
benchmark. Approximate 3D visualization boxes are not exported as measured 3D
predictions; their 3D fields use the official sentinel values.

The global detector confidence remains the fallback, while repeatable
model-class overrides make Car and Pedestrian filtering independent. Use the
actual detector class name: `person` for a COCO checkpoint and `pedestrian` for
a KITTI fine-tuned checkpoint.

```powershell
python track_sequence.py `
  --dataset-root "C:\path\to\KITTI\tracking" `
  --sequence 0000 `
  --model yolo26s.pt `
  --imgsz 640 `
  --yolo-end2end `
  --headless `
  --confidence 0.25 `
  --class-confidence person=0.45 `
  --class-confidence car=0.30 `
  --export-kitti data/kitti_predictions/0000.txt
```

The lowest configured class threshold is passed to YOLO so that an override
below the global value can still recover candidates. Each class-specific value
is then applied before detections update Deep SORT. Invalid or unknown model
class names fail before sequence processing starts.

YOLO26s end-to-end is the selected pretrained control: it was fastest and had

### Reproducible fixed-split exports

Use `export_kitti_tracking_split.py` for multi-sequence work. It starts each
sequence in a fresh process, writes `experiment_manifest.json` atomically, and
records the runtime, model hash, complete configuration, per-sequence command,
duration, status, and output hash. Manifest schema 3 also records the exact
association and tracked-class policies, Deep SORT embedding batch size,
project-code hashes, and Python, Ultralytics, Torch, Deep SORT, NumPy, and
OpenCV versions. Resume is rejected if any of those inputs changed.
`--log-child-output` stores one log per attempt under
`OUTPUT_DIR/logs`, which keeps long runs readable.

For KITTI export, only detector classes that can become evaluated output
(`car`, `person`, or `pedestrian`) enter Deep SORT; exact class gating makes
this equivalent for those tracks while avoiding unused embeddings. The
MobileNet embedder runs under inference mode in batches of four by default.
Record or override that limit with `--embedder-batch-size`; smaller values use
less peak memory. On the current memory-constrained Windows workstation, use
`--embedder-batch-size 1` for long split exports. A standalone batch-4 stress
run completed sequence `0001`, but the subsequent full-tune attempt hit
system-wide virtual-memory pressure on that sequence, so batch 1 is the
reproducible experiment setting here. This changes embedding throughput, not
the detector, class policy, or matching thresholds. Headless runs stop on the
first failed frame instead of silently continuing with a broken temporal state.

Export all 21 labeled tracking sequences with the selected pretrained control:

```powershell
python export_kitti_tracking_split.py `
  --dataset-root "C:\path\to\KITTI\tracking" `
  --output-dir data/tracking_experiments/yolo26s_control_all `
  --model yolo26s.pt `
  --split-preset all `
  --confidence 0.28 `
  --imgsz 640 `
  --yolo-end2end `
  --embedder-batch-size 1 `
  --log-child-output
```

For development, use `--split-preset trackeval_tune`; reserve
`--split-preset trackeval_val` for the locked comparison. Give each preset and
model a different `--output-dir`. The short `smoke_tune` and `smoke_val`
presets test plumbing only and are not evidence for model selection.

For example, after freezing the configuration, the validation export is a new
experiment rather than a resume of the tune export:

```powershell
python export_kitti_tracking_split.py `
  --dataset-root "C:\path\to\KITTI\tracking" `
  --output-dir data/tracking_experiments/yolo26s_control_val `
  --model yolo26s.pt `
  --split-preset trackeval_val `
  --confidence 0.28 `
  --imgsz 640 `
  --yolo-end2end `
  --embedder-batch-size 1 `
  --log-child-output
```

If an export is interrupted, rerun the exactly matching experiment with
`--resume`. Completed outputs are skipped only after their recorded SHA-256
hashes are verified:

```powershell
python export_kitti_tracking_split.py `
  --dataset-root "C:\path\to\KITTI\tracking" `
  --output-dir data/tracking_experiments/yolo26s_control_all `
  --model yolo26s.pt `
  --split-preset all `
  --confidence 0.28 `
  --imgsz 640 `
  --yolo-end2end `
  --embedder-batch-size 1 `
  --log-child-output `
  --resume
```

An existing output directory is otherwise rejected, and a resume with a
different model, image size, threshold, head, tracker setting, embedding batch
size, code hash, runtime, or sequence list is rejected. Earlier interrupted or
pre-class-gating exports remain diagnostic artifacts, not final evidence.

```powershell
python evaluate_kitti_tracking.py `
  --dataset-root "C:\path\to\KITTI\tracking" `
  --predictions data/kitti_predictions `
  --sequences 0000 0012 `
  --tracker-name yolo_deepsort `
  --output-dir data/evaluation
```

The script validates frames, IDs, boxes, confidence fields, ground truth, and
sequence lengths before starting TrackEval. It stages the selected small text
files in a temporary directory; it does not modify the KITTI dataset on `H:`.
Tracker IDs are remapped densely only in that temporary copy, preserving each
identity across frames while preventing excessive memory use from sparse IDs.
Outputs are written under `data/evaluation/<tracker-name>/`:

- `metrics.json`: compact combined and per-sequence metrics;
- `car_summary.txt` and `pedestrian_summary.txt`: official summary tables;
- `car_detailed.csv` and `pedestrian_detailed.csv`: official per-sequence data.

HOTA measures combined detection and association quality, DetA measures
detection accuracy, AssA measures association accuracy, MOTA penalizes false
positives, missed detections, and ID switches, and IDF1 measures identity
consistency. Rates in `metrics.json` are percentages; counts such as FP, FN,
and IDSW are integers. Car and Pedestrian are reported separately because the
official KITTI adapter does not define a combined-class score.

KITTI preprocessing also handles `Van`, seated-person distractors, truncation,
occlusion, `DontCare` regions, and unmatched predictions no taller than 25
pixels. Therefore the post-preprocessing detection counts in `metrics.json`
can be lower than the number of selected-class rows in the input file. See the
[official KITTI adapter](https://github.com/JonathonLuiten/TrackEval/blob/12c8791b303e0a0b50f753af204249e622d0281a/trackeval/datasets/kitti_2d_box.py).

Local evaluation is available only for labeled training sequences. Results on
one sequence or a held-out subset are development diagnostics and must not be
presented as official KITTI test-server benchmark results.

### Fixed-split confidence experiments

KITTI publishes training and test sets, but no official validation split. This
repository records TrackEval's companion development split in
`config/kitti_tracking_splits.json`: 12 training-minus-validation sequences for
tuning and 9 disjoint sequences for validation. Call it `trackeval_val`, not an
official KITTI validation set.

Existing scored prediction files can be screened without rerunning the detector:

```powershell
python tune_kitti_thresholds.py `
  --dataset-root "C:\path\to\KITTI\tracking" `
  --predictions data/kitti_predictions `
  --split-preset smoke `
  --class pedestrian `
  --source-class-confidence 0.28 `
  --output-dir data/threshold_sweeps
```

For predictions from an eight-class KITTI fine-tuned checkpoint, identify the
detector label explicitly:

```powershell
python tune_kitti_thresholds.py `
  --dataset-root "C:\path\to\KITTI\tracking" `
  --predictions data/tracking_experiments/kitti_yolo26s_source `
  --split-preset trackeval `
  --class pedestrian `
  --model-class pedestrian `
  --source-class-confidence 0.28 `
  --output-dir data/threshold_sweeps/kitti_yolo26s_pedestrian
```

The command reads and evaluates **only** tune sequences. It selects by HOTA,
using IDF1, MOTA, and then the lower threshold as deterministic tie-breakers;
records tune input SHA-256 hashes and the TrackEval revision in `sweep.json`;
and writes a compact `sweep.csv`. It neither requires nor discovers holdout
prediction files, and the report marks locked evaluation as `not_run`.

When the prediction directory contains `experiment_manifest.json`, the sweep
also authenticates prediction hashes, detector provenance, image size, YOLO26
head, association policy, tracker configuration, and the effective source
threshold for the selected detector class. Plain older directories remain
supported with an explicit unauthenticated-provenance warning. Candidates
below the recorded effective class threshold are rejected because those
detections were never exported and cannot be recovered post-hoc.

This is a fast post-hoc screen, not a final tracker result: removing rows after
tracking cannot reproduce the Deep SORT state that would result from filtering
detections beforehand. Always rerun the full sequence with the report's
selected value: `--class-confidence person=<selected>` for a COCO checkpoint,
or `--class-confidence pedestrian=<selected>` for a KITTI fine-tuned
checkpoint, and evaluate that new export.
Only after that real rerun is frozen should `evaluate_kitti_tracking.py` read
the separate `trackeval_val` export. The threshold sweep is never the locked
evaluator.
Use the complete split from the JSON configuration for reportable development
results by replacing `smoke` with `trackeval`; `0000`/`0014` is only a short
pipeline smoke test. Explicit disjoint `--tune-sequences` and
`--holdout-sequences` remain available for custom experiments.

The completed tune-only screens selected the existing 0.28 threshold for both
`car` and `person`, so they do not justify changing the control defaults. In an
earlier smoke test on `0014`, changing Deep SORT `n_init` from 3 to 1 increased
Pedestrian HOTA from 17.521 to 18.881 and reduced FN from 99 to 95, but
increased FP from 38 to 47 and reduced MOTA/IDF1. A post-hoc confidence of 0.80
also failed when rerun through the real tracker because it removed every
Pedestrian detection. Because sequence `0014` was inspected repeatedly before
the frozen run, the completed `trackeval_val` result is useful internal
development evidence but is not a pristine holdout.
Experiment outputs are not silently reused; choose a new `--output-dir`, or use
`--overwrite` only when intentional. Prediction and output paths must also be
separate so a sweep cannot rewrite its own source evidence.

Use `--help` on any script for configurable paths, thresholds, device, tracker
settings, and output options.

Interactive controls:

- `N`, Space, Enter, or Right Arrow: next frame
- `B`, `P`, or Left Arrow: previous cached frame
- `S`: save the current render
- `Q` or Escape: quit

Frames are processed by Deep SORT at most once and only in forward order.
Browsing backward or saving cannot accidentally update the tracker again.

## Recommended tracking dataset

Use the official [KITTI Tracking benchmark](https://www.cvlibs.net/datasets/kitti/eval_tracking.php)
first because it has the lowest migration cost for this code. Download at least:

- left color images;
- Velodyne point clouds;
- camera calibration;
- training labels.

After extracting one sequence, the relevant structure is:

```text
KITTI_TRACKING/
  training/
    image_02/0000/*.png
    velodyne/0000/*.bin
    calib/0000.txt
    label_02/0000.txt
```

The local labeled tracking set audit covers all 21 sequences (`0000` through
`0020`): 8,008 camera images, 8,004 LiDAR scans, and 65,301 annotation rows.
The fixed TrackEval development split contains 5,027 tune frames and 2,981
validation frames. Sequence `0001` has camera frames 177--180 but no matching
LiDAR files. Those same four scans are absent from the official tracking
Velodyne archive, so this is a known source-data gap rather than an incomplete
local download.

The preferred interface selects a complete sequence from the dataset root:

```powershell
python track_sequence.py `
  --dataset-root KITTI_TRACKING `
  --sequence 0000 `
  --split training `
  --model yolo26s.pt `
  --imgsz 640 `
  --yolo-end2end

python track_3d_visualization.py `
  --dataset-root KITTI_TRACKING `
  --sequence 0000 `
  --split training `
  --model yolo26s.pt `
  --imgsz 640 `
  --yolo-end2end
```

`--split` defaults to `training`. The dataset root may be either the directory
that contains `training`/`testing` or the split directory itself. Sequence IDs
such as `0` are normalized to `0000`. The loader validates sequence filenames
and cross-modality frame pairing, loads sensor data lazily, caches the shared
sequence calibration, and attaches labels to their corresponding frames.

Missing LiDAR scans are reported per frame without aborting the sequence. The
3D runner still advances YOLO and Deep SORT for that camera frame, but produces
no LiDAR-supported position unless the explicit ray fallback is enabled. This
is the expected behavior for `0001` frames 177--180; camera-only 2D export and
TrackEval remain usable for those frames.

Manual path mode remains available when the files are arranged differently:

```powershell
python track_sequence.py `
  --image-dir KITTI_TRACKING/training/image_02/0000

python track_3d_visualization.py `
  --image-dir KITTI_TRACKING/training/image_02/0000 `
  --pointcloud-dir KITTI_TRACKING/training/velodyne/0000 `
  --calib-file KITTI_TRACKING/training/calib/0000.txt
```

The calibration loader accepts both the KITTI object names (`R0_rect`,
`Tr_velo_to_cam`) and KITTI tracking aliases (`R_rect`, `Tr_velo_cam`).

### Tracking labels

Each row in `training/label_02/<sequence>.txt` is parsed as:

```text
frame track_id type truncated occluded alpha left top right bottom
height width length x y z rotation_y [score]
```

The 17-field form is KITTI ground truth; an optional eighteenth `score` field
is accepted for prediction/result files. Dimensions retain KITTI's
`(height, width, length)` order, locations are `(x, y, z)` in rectified camera
coordinates, and `DontCare` rows are preserved but can be excluded from frame
queries.

Use the sequence loader for synchronized paths, labels, and lazy sensor data:

```python
from utils.kitti_tracking_dataset import KittiTrackingDataset

dataset = KittiTrackingDataset("KITTI_TRACKING", sequence="0000")
frame = dataset[0]
objects = dataset.labels_for_frame(frame.frame_index, include_dont_care=False)
sample = dataset.load_sample(0)  # Loads the image and available LiDAR scan.
```

Or use the parser directly when only annotations are needed:

```python
from utils.kitti_tracking_labels import load_kitti_tracking_labels

labels = load_kitti_tracking_labels("KITTI_TRACKING/training/label_02/0000.txt")
frame_42 = labels.by_frame(42, include_dont_care=False)
track_7 = labels.by_track(7)
track_ids = labels.track_ids
```

Use a held-out subset of the labeled training sequences for student/project
evaluation. Read KITTI's registration, submission, and non-commercial license
conditions before downloading or redistributing the data.

## Fine-tuning YOLO11s and YOLO26s on KITTI

Detector fine-tuning uses the Ultralytics KITTI **object-detection** dataset,
which is separate from the temporal tracking dataset above. The prepared local
root should have 5,985 training images and 1,496 validation images in this
layout:

```text
C:\path\to\Ultralytics\kitti\
  images\train\*.png
  images\val\*.png
  labels\train\*.txt
  labels\val\*.txt
```

The local root used for the documented experiment was verified
against that expected layout: it contains exactly 5,985 train and 1,496
validation images, the same numbers of paired label files, and no unmatched
stems. Ultralytics successfully scanned every image in both splits with zero
corrupt files. This verifies local integrity and layout; it does not replace
the content fingerprints written into each training run's manifest.

Its eight detector classes are, in order: `car`, `van`, `truck`, `pedestrian`,
`person_sitting`, `cyclist`, `tram`, and `misc`. This explains an important
class-name difference: a COCO checkpoint calls the human class `person`, while
the KITTI-trained output calls it `pedestrian`. The result exporter accepts
both and maps them to KITTI Tracking `Pedestrian`.

Train YOLO26s with the direct launcher on a CUDA GPU:

```powershell
python train_kitti_detector.py `
  --model yolo26s.pt `
  --dataset-root "C:\path\to\Ultralytics\kitti" `
  --epochs 100 `
  --imgsz 640 `
  --batch 16 `
  --device 0 `
  --project runs/detect `
  --name kitti_yolo26s
```

On the tested Windows AMD laptop, use the ROCm launcher. Its conservative
defaults are batch 2, zero data-loader workers, physical GPU 0, and logical
Ultralytics device 0 after GPU isolation:

```powershell
.\scripts\train_kitti_rocm.ps1 -Name kitti_yolo26s_rocm
```

Paths and training values are parameters, so a one-epoch custom-subset smoke
run can be requested without editing the script:

```powershell
.\scripts\train_kitti_rocm.ps1 `
  -DatasetRoot "C:\path\to\Ultralytics\kitti_rocm_smoke" `
  -Name rocm_yolo26s_smoke `
  -Epochs 1 `
  -Batch 2 `
  -AllowCustomDatasetSize
```

Run the controlled YOLO11s comparison with the same data and hyperparameters:

```powershell
python train_kitti_detector.py `
  --model yolo11s.pt `
  --dataset-root "C:\path\to\Ultralytics\kitti" `
  --epochs 100 `
  --imgsz 640 `
  --batch 16 `
  --device 0 `
  --project runs/detect `
  --name kitti_yolo11s
```

With the documented explicit `--dataset-root`, the launcher validates all four
image/label split directories, one-to-one stem pairing, and the official
5,985/1,496 split sizes before training. A deliberately smaller KITTI-format
experiment must opt in with
`--allow-custom-dataset-size`. It writes a run-local absolute-path
`kitti_dataset.yaml` and records the exact split counts, byte totals, content
fingerprints, software/hardware environment, and initialization-checkpoint
SHA-256 in `training_request.json`. Image contents are included in the
inventory fingerprint, so a missing or replaced image is detected before an
interrupted run can resume.

Use a new `--name` for every independent run so an existing experiment is
never silently overwritten. The expected selected checkpoints are
`runs\detect\kitti_yolo26s\weights\best.pt` and
`runs\detect\kitti_yolo11s\weights\best.pt`.

If a run is interrupted, repeat its original command unchanged and add
`--resume`, for example:

```powershell
python train_kitti_detector.py `
  --model yolo26s.pt `
  --dataset-root "C:\path\to\Ultralytics\kitti" `
  --epochs 100 `
  --imgsz 640 `
  --batch 16 `
  --device 0 `
  --project runs/detect `
  --name kitti_yolo26s `
  --resume
```

For an AMD run created by the PowerShell launcher, use its matching switch:

```powershell
.\scripts\train_kitti_rocm.ps1 -Name kitti_yolo26s_rocm -Resume
```

Resume is accepted only when that run's `training_request.json`, immutable
configuration, model choice, run-local dataset YAML, and current dataset
fingerprint all match. It then loads only
`runs\detect\kitti_yolo26s\weights\last.pt`, records that checkpoint's SHA-256,
and lets Ultralytics restore the optimizer and epoch state. Missing or changed
inputs are refused; do not use `best.pt` to resume optimizer state.

A CUDA or ROCm GPU is strongly recommended. Native-Windows ROCm was verified
end to end on the RX 7600S with a 32-train/8-validation-image subset: YOLO26s,
one epoch, image size 640, batch 2, and AMP enabled. Training and validation
completed in about 219 seconds and produced both `best.pt` and `last.pt`.
Validation reported mAP50 0.194 and mAP50-95 0.132, but those values are only a
toolchain smoke result from eight validation images and must not be used as an
accuracy comparison.

The first ROCm batch spent about 199 seconds compiling and caching kernels;
later batches ran at roughly 4-5 iterations per second. This one-time JIT delay
is expected for a new shape/cache. The full 100-epoch YOLO26s/YOLO11s comparison
has not been started yet. Accelerator selectors fail when neither CUDA nor ROCm
is available. `--device mps` is accepted only when PyTorch reports MPS support.
An explicit `--device cpu` always requires `--allow-cpu-training`, even on an
accelerator host. That CPU option exists for deliberate smoke tests, but full
training on CPU will be very slow and should not be used for timing comparisons.

Start a fine-tuned tune-split export like this:

```powershell
python export_kitti_tracking_split.py `
  --dataset-root "C:\path\to\KITTI\tracking" `
  --output-dir data/tracking_experiments/kitti_yolo26s_tune `
  --model "runs\detect\kitti_yolo26s\weights\best.pt" `
  --split-preset trackeval_tune `
  --confidence 0.28 `
  --imgsz 640 `
  --yolo-end2end `
  --device 0 `
  --embedder-gpu `
  --log-child-output
```

The remaining evaluation workflow is:

1. Fine-tune YOLO11s and YOLO26s with the same training split, image size,
   epochs, seed, and batch policy.
2. Compare checkpoints and YOLO26 head choice only on `trackeval_tune`; freeze
   one detector, head, image size, and Deep SORT configuration before touching
   `trackeval_val`.
3. Export low-source-confidence predictions for the required split with
   `export_kitti_tracking_split.py`, preserving its manifest and logs.
4. Screen Car and Pedestrian thresholds on tune predictions. Pass
   `--model-class pedestrian` for the fine-tuned checkpoint.
5. Rerun YOLO and Deep SORT into a fresh output directory with each selected
   `--class-confidence`; post-hoc row filtering is not a final tracker run.
6. Evaluate the rerun on the tune split, lock every setting, export
   `trackeval_val`, and evaluate that validation split once.
7. Report Car and Pedestrian HOTA, DetA, AssA, MOTA, IDF1, FP, FN, and IDSW,
   together with the detector checkpoint hash and experiment manifest. Do not
   describe these local development scores as official KITTI test results.

## Current limitations and next work

- The selected YOLO26s control and YOLO11s comparison are pretrained COCO 2D
  detectors, not learned 3D detectors. The KITTI fine-tuning path now works on
  the RX 7600S through ROCm, but only a tiny one-epoch execution smoke run is
  complete; full controlled training is still required.
- The pretrained control now has complete tune and development-validation
  exports and metrics. Pedestrian performance remains substantially weaker
  than Car, so the next detector work is the controlled YOLO11s/YOLO26s KITTI
  fine-tuning comparison on the verified ROCm environment (or a CUDA GPU).
- The nine-sequence development-validation result is not a pristine holdout
  because `0014` was inspected during earlier smoke experiments. It must not be
  presented as an unbiased final result or as an official KITTI server score.
- LiDAR localization uses projected points plus lightweight depth clustering;
  overlapping boxes can associate with the wrong foreground surface.
- Dimensions are class priors and every visualized box currently has zero yaw.
- Deep SORT tracks image appearance and 2D motion, not a Kalman state in 3D.
- Automatic 2D HOTA/IDF1/MOTA evaluation is available, but measured 3D box
  prediction and quantitative 3D IoU evaluation still need to be added.
- Post-hoc score sweeps are candidate screens only; every selected threshold
  must be rerun through YOLO and Deep SORT before it can be compared fairly.
- Ego-motion compensation is required before comparing LiDAR-frame positions
  across moving-vehicle frames.
- The unrelated legacy PowerPoint and stale Office lock file are not part of
  the pipeline and should be replaced/removed before publishing the project.

For a modern second backend after KITTI works, use the official nuScenes mini
split to develop an adapter, followed by its train/validation splits for real
evaluation. That migration requires token-based records, ego poses, and
quaternion transforms rather than KITTI text files.
